## 1. 引导合理性

### 1.1 做得好的地方

- **六步闭环清晰**：PVE → 商店单位 → 商店道具 → 背包整理 → 封盘 → 层结算，与 GDD 单层循环一致。
- **路径与商店关系有说明**：第 2 步提示「第一层第二行左侧格固定连通商店」，与 `pve.js` 中 `floor === 1` 时对 `grid[1][0]` 强制为商店的逻辑一致，降低迷路成本。
- **可跳过**：`跳过引导` 满足老玩家需求。
- **高亮目标**：探索 / 封盘等步骤对按钮加 `guided-target`，方向明确。

### 1.2 问题与风险（P0/P1，首轮试玩记录）

> **说明**：下表为当日上午试玩结论；**§0.1 所列项已在当日下午通过代码修复**，保留作溯源。

| 等级 | 现象 | 影响 |
|------|------|------|
| **P0** | 第 1 步要求立刻「探索节点」，但开局**棋盘上无单位**，本次实测日志为 `战力0 vs 490`，探索**大败**并扣大量 HP。 | 新玩家第一印象是「被碾压」，与「教学关」心理预期冲突；同时与左侧 **稳健策略文案**（先购买前排 → 再探索）**顺序相反**，认知打架。 |
| **P0** | 第 4 步高亮 `#autoArrangeBtn`，但选中商店节点时侧栏默认在 **商店 Tab**；`整理背包` 按钮在 **背包面板内且不可见**。Playwright 点击该按钮 **30s 超时**，真人用户同样需**自己切到「背包」Tab**。 | 引导步骤在 UI 上**不可直接完成**，属于阻断级体验问题。 |
| **P1** | 第 5 步文案写「结束**当前**阶段并封盘」，主界面按钮为「结束阶段并封盘」（`index.html`），**措辞不一致**。 | 轻微困惑，建议统一 copy。 |
| **P1** | 引导第 3 步用 `targetSelector: [data-buy-item-index]` 高亮购买按钮，但未强制切 Tab；若用户停留在背包页，可能看不到高亮目标。 | 与商店页可见性强绑定，建议与步骤 2 一样保证侧栏处于商店。 |

### 1.3 引导层建议（摘要）

1. **调整步骤顺序或首步前置条件**：例如先指向「选商店 → 买 1 个单位 → 再探索」，或开局赠送 1 个基础单位/占位棋子，避免战力为 0 的首次战斗。
2. **进入第 4 步时 `state.ui.sideTab = 'bag'` 并 `render`**（或与 `renderTutorial` 联动自动切 Tab），保证「整理背包」可见。
3. 统一「结束阶段」相关文案与按钮字面。

## 2. 可玩性（从本次流程观察）

- **循环可跑通**：探索、商店购买、背包整理、封盘、异步 PVP（mock）、失败归因卡、保底回退均能触发，**系统深度有展示**。
- **首次战斗挫败感强**：在引导强制路径下仍可能 0 战力进场，**学习曲线过陡**，玩家尚未理解「自走棋要先有人」已被扣血。
- **异步结算体感**：本次出现 **连续两次「异步匹配与结算超时」** 后走保底小负，**节奏被打断**；与文档中「终局层长时间停留」类问题同源，第一层即暴露会降低信任感。
- **归因与建议可用**：日志侧「失败归因卡」与建议条目与当前状态（上阵少、缺前排）相符，**可解释性方向正确**。

## 3. 数值合理性

### 3.1 本次实测关键点

- **初始资源**：默认落点「贮藏塔基」`+6` 金（`runDraft.config.js`），与 `initialGold: 12` 合计开局 **18 金**，购买 T3（4g）+ 道具（2g）仍有余钱，**商店阶段数值尚可**。
- **零阵容 PVE**：`combat.js` 中玩家侧无单位时战力为 0，敌方可仍为多单位阵容，导致 **HP 惩罚区间上限** 被吃到（日志 `HP-24` 量级），对「第一格必为战斗」的新手不友好。
- **层内回复与 PVP 扣血**：层结算有 HP 回复设计；若 PVP 多次失败 + PVE 大亏，前期 HP 会快速离开满血区，**与「第一层教学」预期仍偏硬**。

### 3.2 与既有文档的对照

- [Demo二次试玩复测-2026-03-19.md](./Demo二次试玩复测-2026-03-19.md) 已指出中后期**经济偏快、血线后期偏轻**等问题；本次补充：**前期在引导顺序不当时，血线可以一下被打穿**，与前中期曲线需分开调。

### 3.3 数值向建议

1. **首层首战保护**：例如第一层首次探索固定弱怪、或 `teamPower` 下限、或强制至少 N 个占位单位再允许探索（与引导步骤联动）。
2. **审视「战斗格起手」与空阵容的惩罚上限**：避免单次探索在非 Boss 格即造成 `HP-20+` 的毁灭感。
3. 继续跟进 **mock PVP 超时阈值** 与 UI 等待表现，避免「双超时 + 保底」成为常态路径（参见 [项目文档总览](./项目文档总览.md) 第 4 节待办）。

## 5. 布局合理性

### 5.1 桌面宽屏（本次 1400px）

- **三栏网格**（`styles.css` `.layout`：`0.95fr / 2.2fr / 1fr`，`max-width: 1680px`）：左路径与操作、中棋盘、右背包/日志，**信息分区符合心流**。
- **顶栏 HUD**：`11` 个 `hud-card` 单行 `grid` + `overflow-x: auto`，指标多时**依赖横向滚动**，宽屏下尚可，略挤。
- **左列**：路径金字塔 + 操作区 + 折叠「说明」，**密度较高但可接受**。

### 5.2 窄屏与可访问性

- **`@media (max-width: 1180px)`**：主布局改为**单列**，HUD 改为每行 3 列，**棋盘与侧栏纵向堆叠**，长页面滚动量增加；需实机确认「拖动上阵」「路径点击」在触摸设备上的命中区（`.route-node`、棋格）。
- **层叠**：引导层、开局模态、战斗回放均为全屏/高 `z-index`，需保证**同时只暴露一个主交互层**（本次可见文本中曾出现多段叠在一起的 DOM 文案，若以屏幕阅读器或 `innerText` 采集可能混淆；建议关闭模态时移出焦点或 `aria-hidden`）。

### 5.3 布局向建议

1. HUD 考虑合并次要字段（如齿轮/秘能）或二级折叠，减少首屏横向滚动。
2. 单列布局下将 **「当前阶段可操作按钮」** 固定在视口底部或重复一份，减少上下往返。
3. 引导出现时，对**被遮挡的高亮按钮**做滚动容器内 `scrollIntoView`（若未来布局更复杂）。

## 6. 结论

首轮试玩暴露出 **「先探索导致 0 战力惨败」**、**「整理背包步骤在错误 Tab」** 与 **「PVP 回放被误计入超时竞态」** 等问题；**§0.1 已在代码层闭环**。后续仍可迭代：HUD 信息密度、窄屏固定操作区、引导高亮 `scrollIntoView`、以及教学战失败时的兜底提示（当前轻装战仍以高胜率为目标）。
