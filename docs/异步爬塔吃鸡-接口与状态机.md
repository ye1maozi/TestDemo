# 《塔陨：异步终局》接口与状态机文档

## 1. 文档目标
- 给客户端、服务端、战斗服、数据分析同一份实现基线。
- 明确弱联网异步对局的状态流转、接口契约、幂等与容错策略。
- 作为 MVP（12~16周）开发和联调用文档。

## 2. 对局总览
- 模式：30人同 `RunID`，共享种子与时间窗。
- 结构：层内 PVE（本地）-> 封盘提交快照 -> 异步 PVP（服务端权威）-> 层结算。
- 弱联网：只在关键节点联网；断线允许继续层内操作，但封盘前必须提交。

## 3. 核心实体模型

### 3.1 Run（对局）
- `runId: string`
- `seasonId: string`
- `seed: int64`
- `status: enum` (`RECRUITING | RUNNING | FINISHED | CANCELED`)
- `currentFloor: int`
- `floorDeadlineTs: int64`
- `ruleSetId: string`

### 3.2 PlayerRun（玩家对局态）
- `uid: string`
- `runId: string`
- `hp: int`
- `lifeBadge: int`  // 徽记
- `rank: int?`
- `isEliminated: bool`
- `econGold: int`
- `econGear: int`
- `econMana: int`
- `streak: int`  // 正负表示连胜/连败
- `lastSnapshotId: string?`
- `floorProgress: enum` (`PVE | SNAPSHOT_LOCKED | PVP_DONE | SETTLED`)

### 3.3 Snapshot（构筑快照）
- `snapshotId: string`
- `runId: string`
- `uid: string`
- `floor: int`
- `submittedAt: int64`
- `expiresAt: int64`
- `units: UnitState[]`

- `backpack: BackpackState`
- `relics: RelicState[]`
- `runtimeBuffs: BuffState[]`
- `clientActionHash: string`
- `clientVersion: string`

### 3.4 BattleResult（异步对战结果）
- `battleId: string`
- `runId: string`
- `floor: int`

- `defenderUid: string`
- `winnerUid: string`
- `rounds: int`

- `simVersion: string`
- `createdAt: int64`

## 4. 状态机

### 4.1 Run 状态机
- `RECRUITING` -> `RUNNING`：人数满30或报名超时。
- `RUNNING` -> `FINISHED`：仅剩1人未淘汰或达到终局结算条件。
- `RUNNING` -> `CANCELED`：系统异常回滚（极少见，需补偿）。

### 4.2 Floor 子状态机（每层）
- `PVE_OPEN`：允许探索、商店、背包构筑。
- `PVE_CLOSING`：路径收缩，部分节点关闭。
- `SNAPSHOT_WINDOW`：封盘，允许提交快照。
- `MATCHING`：服务端匹配镜像对手。
- `SIMULATING`：服务端战斗重算。
- `FLOOR_SETTLEMENT`：发奖、扣血/扣徽记、晋层或淘汰。

标准流转：
`PVE_OPEN -> PVE_CLOSING -> SNAPSHOT_WINDOW -> MATCHING -> SIMULATING -> FLOOR_SETTLEMENT -> NextFloor(PVE_OPEN)`

异常流转：
- 未提交快照：`SNAPSHOT_WINDOW` 结束后进入惩罚结算（扣徽记，可直接淘汰）。
- 匹配超时：走补位快照池；仍失败则按 PVE 保底结算（低收益+扣时延分）。

### 4.3 PlayerRun 状态机
- `ALIVE` -> `DOWN_BADGE`：层结算失败或超时。
- `DOWN_BADGE` -> `ELIMINATED`：`lifeBadge <= 0`。
- `ALIVE` -> `FINISHER`：终局胜者/存活结算玩家。

## 5. 时间窗与“缩圈”定义
- 采用“时间圈 + 路线圈”：
  - `T1` 自由探索：170s
  - `T2` 路径收缩：35s
  - `T3` 封盘提交：15s
- `T3` 结束未提交：`lifeBadge -1`，并判定本层战斗失败。
- 路线圈在 `T2` 开始逐步关闭高收益分支，强制向结算节点收束。

## 6. 接口清单（HTTP/JSON 示例）

## 6.1 `POST /v1/run/create-or-join`
用途：创建或加入房间。

请求：
```json
{
  "uid": "u_10001",
  "seasonId": "s1",
  "mode": "ASYNC_30",
  "clientVersion": "0.2.1"
}
```

响应：
```json
{
  "runId": "run_abc",
  "status": "RECRUITING",
  "joinedCount": 18,
  "startDeadlineTs": 1770000000
}
```

## 6.2 `GET /v1/run/config?runId=...`
用途：拉取对局配置、种子、当前层时间窗。

响应关键字段：
- `seed`
- `ruleSetId`
- `currentFloor`
- `phase`
- `phaseDeadlineTs`

## 6.3 `POST /v1/floor/snapshot/submit`
用途：提交封盘快照（幂等）。

请求：
```json
{
  "runId": "run_abc",
  "uid": "u_10001",
  "floor": 3,
  "idempotencyKey": "run_abc_u_10001_f3_v2",
  "snapshot": {
    "units": [],
    "board": {},
    "backpack": {},
    "relics": [],
    "runtimeBuffs": []
  },
  "clientActionHash": "sha256:xxxx"
}
```

响应：
```json
{
  "snapshotId": "snap_123",
  "accepted": true,
  "serverLockTs": 1770000123
}
```

## 6.4 `POST /v1/floor/battle/request`
用途：请求发起异步匹配与战斗结算。

请求：
```json
{
  "runId": "run_abc",
  "uid": "u_10001",
  "floor": 3,
  "snapshotId": "snap_123"
}
```

响应：
```json
{
  "battleTicket": "bt_888",
  "status": "QUEUED"
}
```

## 6.5 `GET /v1/floor/battle/result?battleTicket=...`
用途：轮询战斗结果。

响应：
```json
{
  "status": "DONE",
  "battleId": "b_777",
  "winnerUid": "u_10009",
  "damageToLoserHp": 14,
  "badgeDeltaLoser": 1,
  "replayKey": "rpl_777"
}
```

## 6.6 `POST /v1/floor/settlement/ack`
用途：客户端确认已收层结算，推进下一层。

## 6.7 `GET /v1/run/leaderboard?runId=...`
用途：获取当前存活、排名、淘汰信息。

## 6.8 `POST /v1/run/finish`
用途：对局结束确认与奖励发放。

## 7. 幂等与一致性
- 所有写接口必须带 `idempotencyKey`。
- 快照提交以 `(runId, uid, floor)` 为主键，后到覆盖需版本号更高。
- 战斗结算以 `battleId` 唯一，不允许客户端二次改写。
- 客户端断线重连后先拉 `run/config` 与 `playerRun`，再恢复本地状态。

## 8. 匹配规则（异步 PVP）
- 优先级：
  1. 同层、同时间窗提交者
  2. 同层、战力差在阈值内
  3. 同段位快照池补位
- 避免重复：
  - 同层尽量不重复交手
  - 连续两层不得匹配同一对手（可配置）
- 保护机制：
  - 低徽记玩家短期软保护，避免连续高战力对手

## 9. 服务端战斗重算规范
- 使用固定 `simVersion`（热更受控）。
- 输入仅来自服务端已验签快照。
- 伪随机数流由 `seed + floor + battleId` 派生。
- 输出统一结构：胜负、剩余单位、伤害、触发日志、回放数据。

## 10. 结算伪代码（示意）
```ts
function settleFloor(player, snapshot, floorCtx) {
  if (!snapshot || snapshot.submittedAt > floorCtx.snapshotDeadline) {
    player.lifeBadge -= 1;
    applyTimeoutPenalty(player);
    return buildSettlement("TIMEOUT_LOSE");
  }

  const enemySnap = matchmaking.pickEnemy(snapshot, floorCtx);
  const battle = simulator.run(snapshot, enemySnap, floorCtx.simSeed);

  if (battle.winnerUid !== player.uid) {
    player.hp -= calcHpDamage(battle);
    if (player.hp <= 0) {
      player.lifeBadge -= 1;
      player.hp = respawnHpByRule(player); // 徽记制重置血量
    }
  } else {
    grantWinReward(player, floorCtx);
  }

  if (player.lifeBadge <= 0) {
    player.isEliminated = true;
  }

  return buildSettlementFromBattle(battle, player);
}
```

## 11. 错误码建议
- `RUN_NOT_FOUND`
- `RUN_NOT_IN_PHASE`
- `SNAPSHOT_EXPIRED`
- `SNAPSHOT_SCHEMA_INVALID`
- `BATTLE_QUEUE_TIMEOUT`
- `IDEMPOTENCY_CONFLICT`
- `CLIENT_VERSION_BLOCKED`
- `ANTI_CHEAT_RISK_REJECTED`

## 12. 反作弊落地点（接口层）
- `snapshot/submit`：
  - 校验背包格子合法性与物品唯一性
  - 校验单位星级与经济来源闭环
- `battle/request`：
  - 仅允许引用本人本层有效快照
- `run/finish`：
  - 再次校验关键层日志哈希链
- 高风险处理：
  - 进入隔离匹配池
  - 奖励延迟发放
  - 审核后补发或封禁

## 13. 监控与告警
- 性能：
  - `battle_sim_p95_ms`

  - `idempotency_conflict_rate`

  - `floor_timeout_penalty_rate`

  - `avg_run_duration_sec`

## 14. 联调与测试用例（最小集）
- 正常流：
  - 30人满房 -> 8层 -> 冠军结算
- 异常流：
  - 封盘前断线重连并成功提交
  - 快照重复提交（幂等验证）
  - 匹配超时走补位池
  - 战斗服重启后的结果一致性
- 安全流：
  - 非法快照字段注入
  - 篡改 action hash
  - 重放旧快照跨层提交

## 15. 客户端实现建议
- 本地状态机与服务端 phase 对齐，不做“本地自增楼层”。
- 封盘前 UI 强提示（15秒、5秒、1秒）。
- 提交失败默认重试 3 次，指数退避（300ms/900ms/2700ms）。
- 若 `SNAPSHOT_WINDOW` 结束仍失败，立即请求“兜底上报”接口。

## 16. 版本控制建议
- 接口版本：`/v1`，破坏性变更升 `v2`。
- 模拟器版本：`simVersion` 独立管理，可灰度放量。
- 配置版本：`ruleSetId` + `seasonId` 双键追踪。
